How It Works

A complete algorithmic disclosure of the PDF Accessibility Engine: every stage, every threshold, every confidence model. Open source, free, no vendor lock-in.

1. Overview

The PDF Accessibility Engine is an open-source pipeline that ingests arbitrary PDF documents and produces fully tagged, PDF/UA-1 compliant outputs with embedded fonts, structural semantics, and machine-verified accessibility. It operates as a nine-stage sequential pipeline: each stage reads the output of its predecessor, applies deterministic algorithms with tunable parameters, and emits a confidence-scored intermediate result. No cloud services, no proprietary models, no vendor lock-in.

2. Stage-by-Stage Algorithm Detail

Each subsection below describes one pipeline stage: what it does, the algorithm it implements, the specific parameter values drawn from the codebase, the confidence model that quantifies output quality, and the edge cases the stage is designed to handle.

2.1 Parser (pdfjs-dist)

Back to top

Purpose

Extract every text span, its bounding box, font metadata, and page geometry from the source PDF. When native text is absent or degraded, fall back to multi-pass OCR with consensus scoring.

Algorithm

for each page in document:
    items = page.getTextContent()          // PDF.js text extraction
    blocks = groupByGeometricProximity(items)

    if blocks.length ≤ 1
       OR totalChars ≤ 24
       OR textCoverage ≤ 0.2%:
        blocks = runMultiPassOCR(page)      // OCR fallback triggered

Multi-Pass OCR

The OCR subsystem generates 3 render variants of each page, then processes each variant with 3 recognition profiles, yielding up to 9 candidate transcriptions per page.

Render VariantDPIMode
gray-300300Grayscale
binary-300300Binary (Otsu threshold)
gray-450450Grayscale, higher resolution

Consensus scoring selects the transcription with the highest character-level agreement across candidates. Language detection auto-switches between eng and spa based on trigram frequency analysis.

Confidence Model

Native extraction: confidence 1.0 (deterministic). OCR: confidence equals the consensus ratio among the 9 candidates, typically 0.82–0.97 for clean scans, 0.55–0.75 for degraded originals.

Edge Cases

  • Invisible-text PDFs (pre-OCR'd): native extraction succeeds, OCR skipped
  • Mixed pages (some text native, some scanned): coverage threshold triggers OCR for the full page
  • Rotated text: PDF.js transform matrix decomposed to recover correct bounding boxes

2.2 Layout Analyzer

Back to top

Purpose

Determine the spatial structure of each page: columns, headings, tables, and artifacts (headers, footers, page numbers).

Column Detection

gapThreshold = max(0.16 × pageWidth, 48px)

for each horizontalGap between adjacent text blocks:
    if gap.width ≥ gapThreshold:
        markColumnBoundary(gap.x)
16%Gap threshold (% page width)
48pxMinimum gap (absolute)

Heading Classification

A text block is classified as a heading when its font size exceeds the dominant body font size by a configurable ratio:

  • Standard rule: font-size ratio ≥ 1.55
  • Bold variant: font-size ratio ≥ 1.3 when the block is bold

Heading levels (H1, H2, H3) are assigned by rank-ordering the distinct heading font sizes within the document.

Table Detection (3 Methods)

The analyzer runs three independent table detectors and merges their results:

  1. Vector gridlines: detects explicit horizontal and vertical rules drawn by the PDF content stream.
  2. Text-grid alignment: identifies text spans arranged in a regular grid pattern (cell alignment within tolerance).
  3. Borderless column alignment: finds columns of text that share vertical anchors without any visible rules.

Table Confidence Formula

confidence = min(0.88 + 0.02 × rows + 0.01 × cols, 0.98)
0.88Base confidence
+0.02Per row
+0.01Per column
0.98Ceiling

Borderless Table Detection

A region qualifies as a borderless table when ≥ 60% of candidate rows share at least 2 column anchors (vertical x-coordinates within 3px tolerance).

Confidence Model

Column boundaries: validated by requiring ≥ 3 blocks on each side of a gap. Heading classification: cross-checked against font-size histogram; isolated outliers are demoted to body text.

Edge Cases

  • Single-column legal filings with indented block quotes (not misdetected as columns)
  • Three-column newsletter layouts
  • Tables that span the full page width without any column gaps

2.3 Semantic Engine

Back to top

Purpose

Assign a structural role to every block: P, H1–H3, TH, TD, LI, or Artifact. Group related blocks into composite structures (tables, lists) that may span multiple pages.

Role Assignment

Each block receives the highest-priority role whose conditions are satisfied:

RoleConfidenceCondition
Artifact0.99Header/footer zone, page number pattern, watermark
TH0.96Inside detected table, first row or bold-first-column
TD0.90Inside detected table, non-header cell
Table (composite)0.96Region matched by any table detector
LI0.94Bullet/number prefix + vertical gap ≤ 28px to siblings
H1–H30.93Heading classification from layout analyzer
P0.88Default: body text not matched by any other rule

Table Continuation

Tables frequently span page breaks. The engine uses two heuristics:

  • Same-page continuation: a new table region is merged with the preceding table if the vertical gap (Y) is ≤ 48px or the horizontal offset (X) is ≤ 160px.
  • Cross-page continuation: column anchor positions from the previous page are matched against the new page; if anchors align within tolerance, the table spans the page break.

List Grouping

Consecutive list-item blocks are grouped into a single <L> structure when the vertical gap between items is ≤ 28px.

Edge Cases

  • Nested lists (indentation-based level detection)
  • Tables inside list items
  • Page headers repeated on every page (artifact deduplication)

2.4 Paragraph Merger

Back to top

Purpose

PDF extractors typically emit one text block per visual line. The paragraph merger reconstitutes logical paragraphs by analyzing spatial signals, reducing the number of <P> tags by an order of magnitude while preserving semantic boundaries.

Algorithm: Text-Structure Strategy

// Phase 1: learn document metrics
bodyMargin  = detectDominantLeftMargin(blocks)
medianSpace = computeMedianLineSpacing(blocks)

// Phase 2: scan for break signals
for each consecutive pair (blockA, blockB):
    gap = blockB.top - blockA.bottom

    if gap > 2.2 × medianSpace:       // vertical gap break
        insertParagraphBreak()
    else if blockB.left - bodyMargin > 20px: // first-line indent
        insertParagraphBreak()
    else if columnChanged(blockA, blockB):  // column boundary
        insertParagraphBreak()
    else:
        mergeIntoParagraph()

Validation

Five post-merge validators catch errors before output:

  1. Cross-column validator: rejects merges that span column boundaries
  2. Length validator: flags implausibly long merged paragraphs (> 80 lines)
  3. Embedded heading validator: splits paragraphs that contain a heading-sized line in the middle
  4. Reading order validator: ensures merged paragraph text follows left-to-right, top-to-bottom order
  5. Spacing consistency validator: rejects merges where internal line spacing varies by > 40%
82%P-line reduction
11,797 → 2,095Legal corpus paragraphs
0.06%Error rate
1 / 1,658Errors in merged paragraphs

Confidence Model

Each merged paragraph inherits the minimum confidence of its constituent blocks. Merges flagged by any validator are split and individually re-evaluated.

Edge Cases

  • Legal documents with numbered paragraphs (number treated as paragraph start, not list item)
  • Footnotes with smaller font size (excluded from merge with body text)
  • Block quotes with altered left margin (indent threshold adapts per-column)

2.5 Reading Order

Back to top

Purpose

Establish a single linear sequence for all tagged content that matches the order a sighted reader would follow. This sequence determines the screen reader navigation order.

Algorithm

sort blocks by:
    1. page number                       // ascending
    2. phase: header > content > footer  // semantic zone
    3. column band index                 // left to right
    4. vertical position (top)            // top to bottom
    5. horizontal position (left)         // left to right tie-break

Key Parameters

ParameterValuePurpose
Line-group epsilon6ptBlocks within 6pt vertical tolerance are treated as the same line and sorted left-to-right
Column band thresholdmax(18% horizontal span, 64px)Minimum gap width to establish a column boundary for reading order purposes

Sub-Pixel Jitter Fix

Font metric rounding can shift block baselines by fractions of a point, causing left-column text to sort after right-column text when sorted naively by Y then X. The 6pt line-group epsilon absorbs this jitter, ensuring correct left-to-right reading within a visual line.

Edge Cases

  • Floating sidebars (assigned to nearest column band)
  • Full-width headings above multi-column body text
  • Footnote regions at page bottom (sorted after body content, before footer artifacts)

2.6 Tag Builder

Back to top

Purpose

Construct the PDF structure tree: a hierarchical tag tree that maps every content element to its semantic role per the PDF/UA-1 specification.

Structure Tree

Document
  ├─ Sect
  │   ├─ H1, H2, H3       // headings
  │   ├─ P                 // paragraphs
  │   ├─ L                 // list
  │   │   └─ LI              // list items
  │   └─ Table
  │       ├─ THead
  │       │   └─ TR → TH, TH, TH
  │       └─ TBody
  │           └─ TR → TD, TD, TD
  └─ Sect ...

Table Normalization

Raw table data is normalized into a standards-compliant structure with explicit THead, TBody, and TFoot sections. Irregular tables (rows with varying cell counts) receive placeholder cells to form a rectangular grid.

Header Row Detection

A row is promoted to THead when any of these conditions hold:

  • All cells in the row are tagged TH
  • ≥ 75% of cells in the first 2 rows are TH
  • The entire first row is ALL CAPS text

Row Header Pattern

When column 0 of a table contains cells tagged TH, those cells are preserved as TH within TBody rows (row headers), and receive a Scope="Row" attribute.

Heading Clamping

Heading levels are clamped to H1–H3 by default. This is configurable via the profile system; the maximum level can be raised to H6 if needed.

Edge Cases

  • Tables with merged cells (colspan/rowspan inferred from geometry)
  • Deeply nested structures (Sect within Sect, up to 4 levels)
  • Empty table cells (tagged as TD with empty content, not omitted)

2.7 Font Embedder

Back to top

Purpose

Ensure every font used in the output PDF is fully embedded with a valid ToUnicode mapping, as required by PDF/UA-1. Repair or substitute fonts that are missing, partially embedded, or lack character-to-Unicode mappings.

24-Check Font Analysis

Each font in the source PDF is evaluated across 5 categories with a total of 24 individual checks:

CategoryChecksExamples
Embedding5Font program present, subset valid, encoding declared
ToUnicode6CMap present, ranges parseable, coverage ≥ 95%
Metrics4Widths array matches glyph count, no zero-width visible glyphs
Identity5BaseFont name resolvable, type declared, flags consistent
License4fsType readable, embedding bits permissive

ToUnicode Reconstruction

When a font lacks a ToUnicode CMap, the embedder attempts reconstruction in priority order:

  1. TTF cmap table: parse the TrueType cmap table directly from the embedded font program
  2. Type1 Differences + AGL: map character codes via the Differences array and the Adobe Glyph List
  3. CID ROS: for CIDFont dictionaries, use the Registry-Ordering-Supplement to locate a standard CMap
  4. Synthesis: last resort — build a mapping from observed character codes to the most likely Unicode code points based on glyph shapes and positional context

Plan Emission

After analysis, each font receives one of these plans:

  • embed-as-is — font is healthy, no changes needed
  • inject-to-unicode — embed a reconstructed ToUnicode CMap
  • re-embed-subset — re-subset the font program to fix structural issues
  • substitute-fallback — replace with a Noto fallback font

Fallback Fonts

FallbackCoverageLicense
Noto SansLatin, Cyrillic, GreekSIL OFL 1.1
Noto SerifLatin, Cyrillic, GreekSIL OFL 1.1
Noto Sans MonoLatin, Cyrillic, GreekSIL OFL 1.1
Noto Sans SymbolsSymbols, Dingbats, MathSIL OFL 1.1

License Handling

The fsType field from the OS/2 table is read for every embedded font. If the field indicates restricted embedding, the engine emits a warning but never blocks processing. The warning is surfaced in the font health report for human review.

Edge Cases

  • Fonts with Identity-H encoding but no embedded CMap (common in CJK PDFs)
  • Type3 bitmap fonts (always substituted with Noto fallback)
  • Subset fonts with non-standard glyph names (AGL fallback + heuristic mapping)

2.8 PDF Writer

Back to top

Purpose

Produce the final tagged PDF output. The writer supports two modes: native (preserving the original vector text) and raster (rendering pages as images with an invisible text overlay).

Native Mode

Native mode preserves the original PDF content stream, injecting only the tag structure. This is a three-step process:

  1. Operator-level parser: decomposes each content stream into individual PDF operators (Tj, TJ, Td, Tm, etc.)
  2. Tag matcher: correlates parsed text operators with semantic blocks from the pipeline, achieving a 94% match rate
  3. BDC/MCID/EMC injection: wraps matched operators in marked-content sequences that reference the structure tree
94%Tag match rate (native)
~1.05×File size (native vs. original)
~10×File size (raster vs. original)

Raster Mode

When native tagging is not feasible (complex content streams, encrypted layers, or operator parsing failures), the writer falls back to raster mode:

  • Each page is rendered at 144 DPI as a raster image
  • An invisible text overlay is positioned over the image using the extracted text coordinates
  • The overlay text is tagged with the full structure tree

CIDSet Cleanup

The writer performs CIDSet cleanup via incremental save, ensuring the CIDSet stream accurately reflects the glyphs present in the output. This prevents false positives from veraPDF's font validation.

Table Scope Attributes

Every TH cell is guaranteed a Scope attribute (Column, Row, or Both) as required by PDF/UA-1.

Edge Cases

  • Content streams with inline images interleaved with text operators
  • Pages with multiple content streams (concatenated before parsing)
  • Incremental-update PDFs with conflicting object versions

2.9 Validator

Back to top

Purpose

Verify the output PDF against the PDF/UA-1 specification. The validator runs in two phases: a fast internal font audit, followed by a full veraPDF compliance check.

Phase 1: Pre-veraPDF Font Audit

The internal font audit runs 8 targeted checks that catch approximately 90% of font-related PDF/UA failures before the more expensive veraPDF process starts:

Finding CodeDescription
FONT_NOT_EMBEDDEDFont program missing entirely
TO_UNICODE_MISSINGNo ToUnicode CMap and no reconstructable mapping
TO_UNICODE_INCOMPLETEToUnicode CMap covers < 95% of used glyphs
ENCODING_MISSINGNo declared encoding for a simple font
WIDTHS_MISMATCHWidths array does not match glyph count
SUBSET_INVALIDSubset prefix present but font program not subsetted
CIDSET_MISSINGCIDFont lacks required CIDSet stream
TYPE3_DETECTEDType3 font cannot be embedded per PDF/UA-1
~90%Font failures caught pre-veraPDF
8Finding codes

Phase 2: veraPDF Validation

The output PDF is submitted to veraPDF for full PDF/UA-1 compliance checking. The validator parses the veraPDF XML report and maps findings to human-readable diagnostics.

Resilience

If veraPDF crashes or times out (which can occur with very large or structurally unusual PDFs), the validator emits a VALIDATOR_EXECUTION_FAILED finding and the job completes successfully with a warning. The tagged PDF is still produced; only the compliance verification is degraded.

Edge Cases

  • PDFs with thousands of fonts (batched font audit to limit memory)
  • veraPDF Java heap exhaustion (caught and reported, not propagated)
  • Encrypted PDFs that decrypt successfully but contain damaged font streams

4. Native PDF Preservation

Back to top

What "Native PDF Retained" Means

When the engine reports Native PDF Retained, it means the original PDF content streams — vector text, embedded fonts, graphics, form fields — are preserved byte-for-byte. The only additions are the structure tree (tags), marked-content operators, and any font repairs. The result is a tagged PDF that is visually identical to the original and carries no rasterization artifacts.

3-Component Architecture

  1. Operator Parser: decomposes each page's content stream into an ordered sequence of PDF operators. Handles compressed streams (FlateDecode), inline images, and nested form XObjects.
  2. Tag Matcher: correlates text-drawing operators (Tj, TJ, ', ") with the semantic blocks produced by the pipeline. Uses a combination of text content matching and positional alignment. Achieves a 94% match rate across a diverse corpus.
  3. Content Stream Rewriter: injects /Tag BMC, /MCID BDC, and EMC operators around matched text runs. Unmatched operators are wrapped in /Artifact BMC ... EMC to satisfy PDF/UA's requirement that all content be tagged.

File Size Comparison

604 KBNative tagged output
6,440 KBRaster tagged output
10.7×Size ratio (same legal filing)

Auto Mode

The default auto mode attempts native tagging first. If the operator parser or tag matcher fails on a given page (match rate below threshold or content stream unparseable), that individual page falls back to raster mode. The final output may contain a mix of natively tagged and rasterized pages, optimizing for both fidelity and file size.

5. Corruption Repair

Back to top

Before tagging, the engine performs a comprehensive health assessment of the source PDF. This catches structural and font-level corruption that would cause downstream failures or PDF/UA non-compliance.

8 Structural Checks

  • Cross-reference table integrity
  • Object stream decompression
  • Page tree consistency
  • Content stream parsing (all pages)
  • Resource dictionary completeness
  • Font reference resolution
  • Image stream integrity
  • Annotation and form field validity

24 Font Health Checks

The same 24-check analysis from the Font Embedder (Section 2.7) is applied as a pre-flight diagnostic. Results are aggregated into a per-font and overall health grade.

Health Scoring: A–F Grades

GradeScore RangeInterpretation
A24/24 checks passFully healthy, no intervention needed
B21–23 checks passMinor issues, auto-repaired
C17–20 checks passModerate issues, repair attempted with warnings
D12–16 checks passSignificant issues, fallback fonts likely needed
F< 12 checks passSevere corruption, full font substitution

Grades are computed per font and then aggregated to an overall document grade (the lowest individual font grade). The health report is accessible from the dashboard and is included in the job's diagnostic output.
